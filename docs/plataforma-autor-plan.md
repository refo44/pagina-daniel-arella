# Daniel Arella

## Plan Maestro de la Plataforma de Autor  
**Versión canónica**

La identidad corporativa definida en `identidad-corporativa.md` es una capa visual y editorial ya cerrada.  
Este documento define algo más profundo: el sistema que permite que la obra de Daniel Arella exista, se lea, se recorra y perdure como un cuerpo vivo.

Este sitio no es una web.  
Es un territorio editorial.

El problema que resuelve este plan no es estético.  
Es ontológico:

- ¿Qué es un poema aquí?  
- ¿Qué es un libro?  
- ¿Cómo se atraviesa una obra sin perderla?  

---

## Principios operativos

Los grandes sitios de autor no funcionan como portafolios ni como tiendas.  
Funcionan como espacios vivos de obra.

| Principio                 | Qué significa                                                                 |
|---------------------------|-------------------------------------------------------------------------------|
| **Espacio vivo**          | El sitio cambia, rota y respira. No es un archivo muerto ni una portada fija. |
| **Una sola tesis en el home** | El home responde una sola pregunta: qué obra está viva ahora.               |
| **La marca es atmósfera** | Tipografía, ritmo y silencio pesan más que los ornamentos.                   |
| **Relación directa**      | El lector se conecta con el autor, no con plataformas.                       |
| **Prueba social silenciosa** | Premios y editoriales existen, pero no gritan.                             |

Este plan no copia esos sitios.  
Los convierte en arquitectura.

---

## Las ocho capas

La plataforma está diseñada en capas que no se contaminan.  
Cada una cumple una función precisa.

---

### Capa 1. Identidad

Define cómo se ve y cómo se siente el sistema.

Incluye:

- Tipografía  
- Paleta  
- Ritmo  
- Uso del blanco  
- Oscuridad y luz  
- Jerarquía editorial  
- Voz visual  

Esta capa ya está cerrada en `identidad-corporativa.md`.

---

### Capa 2. Arquitectura de información

Define qué existe.

Un sitio de autor serio necesita:

| Sección         | Función                          |
|-----------------|----------------------------------|
| Inicio          | Presente vivo de la obra         |
| Poesía          | Libros, ciclos, poemas           |
| Ensayo          | Pensamiento, crítica, reflexión  |
| Narrativa       | Relatos, cuentos                 |
| Talleres        | Cursos y actividad pedagógica    |
| Archivo         | Biblioteca completa              |
| Sobre el autor  | Biografía intelectual            |
| Publicaciones   | Libros, PDFs, editoriales        |
| Contacto        | Prensa, editores, lectores       |

El archivo no es un feed.  
Es una biblioteca navegable.

---

### Capa 3. Modelo de contenido

Define qué es cada cosa.

| Tipo     | Qué representa                 |
|----------|---------------------------------|
| Poem     | Un poema individual             |
| Book     | Una obra cerrada (puede reunir poemas, ensayos, relatos) |
| Essay    | Pensamiento                     |
| Story    | Narrativa                       |
| Workshop | Taller o curso                  |

Un poema no es un post. Un libro no es una página. Una obra no es un blog.

Los ciclos o conjuntos de poemas se narran dentro del libro o con enlaces; no son un tipo de contenido aparte. Los fragmentos y textos breves viven como poemas o ensayos según el caso.

Las taxonomías (tema, periodo y, opcionalmente, forma) permiten que la obra sea navegable. El idioma lo gestiona el plugin de traducción.

---

### Capa 4. Gramática de navegación

Define cómo se recorre una obra.

Incluye:

- Breadcrumbs por libro  
- “Del mismo libro”, “Poemas relacionados”, “Ensayos conectados”  

Archivo por: Tema, Periodo, Libro (relación), Tipo; opcionalmente Forma.

El lector siempre sabe dónde está dentro del cuerpo de la obra.

---

### Capa 5. Estrategia de publicación

Aquí Daniel decide.  
El sistema solo lo hace posible.

Incluye:

- Publicar por libros (ciclos narrados en libro cuando aplique)  
- Abrir o cerrar etapas  
- Publicar PDFs  
- Mostrar obra inédita  
- Activar o desactivar secciones  
- Definir qué está vivo ahora en el home  
- Relación por email bajo tono autoral  

El sistema sostiene cualquier ritmo que el autor adopte.

---

### Capa 6. Sistema editorial

Define cómo se escribe para este territorio.

Incluye:

- Longitud  
- Notas  
- Citación  
- Fragmentos  
- Voz  

La plataforma no dicta estilo.  
Solo garantiza coherencia.

---

### Capa 7. Infraestructura

La base invisible que protege la obra:

- Hosting estable  
- Backups  
- SSL  
- Cache  
- Control de versiones  
- Staging  

Una obra sin backups no es una obra.  
Es un riesgo.

---

### Capa 8. Legal y derechos

Protege la obra:

- Copyright  
- Licencias  
- Permisos de descarga  
- Contacto editorial  

---

## Lo que este sistema permite

Sin tocar código, Daniel puede:

- Publicar un poema o un libro  
- Abrir o cerrar ciclos  
- Traducir obras  
- Cambiar de etapa  
- Activar talleres  
- Reordenar su archivo  
- Cambiar el foco del home  

El sistema no se rompe.

---

## Venta y suscripción sin romper el territorio

Este sistema sí permite:

- Vender libros  
- Vender talleres  
- Recibir suscriptores  

Pero nunca como marketing.

Se hace como:

- “Adquirir el libro”  
- “Inscribirse al taller”  
- “Recibir textos”  

La obra es el centro.  
La transacción es un gesto editorial.

---

## Lo que este sistema NO es

No es:

- Un funnel  
- Una landing  
- Una tienda  
- Un CMS inflado  

Es un territorio de obra que puede sostener dinero sin perder dignidad.
